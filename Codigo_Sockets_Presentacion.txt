1.1 La Arquitectura Cliente-Servidor 🏢
Cualquier sistema conectado del mundo se apoya en que alguien Pide (Cliente) y alguien Escucha y Responde
(Servidor).

👨‍💻 El Cliente (Iniciador)
Siempre inicia la conexión hacia la IP + Puerto del servidor.
Duración corta en red (conecta → pide → recibe → cierra).
Ej: Tu navegador, una app de móvil, un cliente de chat.

🖥️ El Servidor (Oyente)
Programa de bucle infinito esperando llamadas en un Puerto fijo.
Procesa peticiones y devuelve respuestas a múltiples clientes.
Ej: Google.com, un servidor de chat, una API REST en producción.

2.2 Funcionamiento del Servidor Java
Para establecer nuestra comunicación fiable TCP, usamos dos clases muy marcadas en el ecosistema Java que
encapsulan toda esa complejidad de la red.

1️⃣ java.net.ServerSocket
Es la "operadora". Se asigna a un puerto del SO con `new ServerSocket(PORT)` .
Llamar a su método `.accept()` hará que el programa bloquee la ejecución (quede congelado) hasta que llegue y se conecte un cliente
físico.

2️⃣ java.net.Socket
Es el tubo de la conexión activa. Lo devuelve la operadora al hacer `accept()` .
Tiene sus propios InputStream y OutputStream acoplados para recibir y enviar los textos o datos de una forma natural.

2.3 Las Dos Clases Maestras de Java 🧬

Las 3 Excepciones Míticas de Red
Inevitables en tus prácticas. Apréndetelas antes de chocar con ellas:

BindException

Address already in use. Has lanzado tu `Servidor.java` dos veces y el puerto OS sigue
ocupado con el proceso anterior. Matar el proceso en el IDE o cambiar el número de puerto.

ConnectException

Connection refused. El cliente intentó llamar pero nadie responde en ese puerto. Verifica que el
servidor está corriendo antes de lanzar el cliente.

SocketException

Connection reset. Estabas leyendo tranquilamente y el otro extremo cerró el socket
abruptamente (red caída, proceso matado). Tu bucle `while` recibirá `null` en
`readLine()` . Controla ese `null` .
